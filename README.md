# GitHub Leaderboard

Static site that ranks GitHub contributors across one or more repositories (or a whole organization) starting from a given date or commit SHA. Data is fetched from the public GitHub API by a Python script that runs in GitHub Actions and publishes the result to GitHub Pages.

## Metrics

| Metric | Description |
|---|---|
| `commits` | Number of commits authored |
| `java_lines` | Lines changed (additions + deletions) on `.java` files |
| `gaml_lines` | Lines changed on `.gaml` files |
| `wiki_lines` | Lines changed on wiki repositories (`*.wiki`) |
| `issues_opened` | Issues opened by the user |
| `issues_closed` | Issues opened by the user that are now closed |
| `prs_opened` | Pull requests opened by the user |
| `prs_merged` | Pull requests authored by the user that were merged |
| `global_score` | Normalized mean of all metrics (0–1), recomputed client-side when the date range is changed |

Every metric is also stored in a monthly `timeline` (`YYYY-MM`) per user, which lets the UI:
- filter by date range (from / to month),
- render a cumulative evolution chart for the top 5 contributors on the selected metric.

## Project structure

```
gama-leaderboard/
├── config.json              # repos / org / start date or commit
├── scripts/build.py         # fetches GitHub API → docs/data.json
├── docs/                    # served by GitHub Pages
│   ├── index.html
│   ├── app.js
│   ├── style.css
│   └── data.json            # generated by the builder (also committed back)
└── .github/workflows/build.yml
```

## Configuration (`config.json`)

- `org`: a GitHub organization name — every public repository will be analyzed.
- `repos`: an explicit list of `owner/repo` (merged with the org list).
- `since`: ISO date (e.g. `2024-01-01T00:00:00Z`) applied to all repos by default.
- `repo_overrides`: per-repo override — the value may be a 40-char commit SHA, which is resolved to that commit's timestamp.
- `exclude_repos`, `exclude_bots`, `max_commits_per_repo`.

Example — scan `gama-platform`, but start from a specific commit on `gama`:

```json
{
  "org": "gama-platform",
  "since": "2024-01-01T00:00:00Z",
  "repo_overrides": {
    "gama-platform/gama": { "since": "abcdef0123456789abcdef0123456789abcdef01" }
  }
}
```

## Local usage

```bash
export GITHUB_TOKEN=ghp_xxx   # strongly recommended (5000 req/h vs 60)
python scripts/build.py
# then open docs/index.html in a browser
```

Without a token, the public API is limited to 60 requests per hour — only usable to test with one or two small repos.

## Publishing on GitHub Pages

> **Important** — the site must be served from `docs/`, NOT from the repository root. Otherwise GitHub Pages will render `README.md` instead of the leaderboard.

1. Push the project to a GitHub repository.
2. Go to **Settings → Pages**.
3. Pick **one** of these two sources:
   - **GitHub Actions** (recommended): the workflow below deploys `docs/` on every run.
   - **Deploy from a branch**: select branch `main`, folder `/docs`. The workflow commits the updated `data.json` back to `main`, so Pages picks it up automatically.
4. The workflow `.github/workflows/build.yml` runs:
   - on every push to `main` (except changes limited to `docs/data.json`, to avoid loops),
   - every day at 06:00 UTC (cron),
   - on demand (`workflow_dispatch`).
5. The `GITHUB_TOKEN` provided by Actions is used automatically — no secret to configure.

Your leaderboard will then be available at `https://<user>.github.io/<repo>/`.

### If the site still shows the README
- Make sure the Pages source is **not** `main / (root)`. It must be `GitHub Actions` or `main / docs`.
- The repository landing page on `github.com/<user>/<repo>` always shows the README — that is normal. The leaderboard lives on the Pages URL (`*.github.io`). Add it to the repository's **About → Website** field so the link is visible from the repo home.

### If `data.json` looks stale
- The workflow commits the refreshed `docs/data.json` back to `main` after each run. If you still see old data, hard-refresh (Ctrl+F5) — the HTML/JS may be cached even though `data.json` is fetched with `cache: "no-store"`.
- Check the latest run in the **Actions** tab. If `deploy` failed, the most common cause is Pages source being set to `Deploy from branch / root` while the workflow tries to use the Actions source — pick one consistent mode.

## Notes & caveats

- Getting per-extension line stats requires fetching each commit individually, so the build cost is roughly **1 API request per commit**. `max_commits_per_repo` (default 2000) caps this.
- GitHub wiki Git repos (`repo.wiki.git`) are not exposed via the REST API. This project instead treats any repo whose name ends with `.wiki` as a wiki (the convention used e.g. by `gama-platform/gama.wiki`).
- `issues_closed` counts issues opened by the user that are currently `closed`, not issues closed *by* that user (not trivially available via the REST API).
- `global_score` is the mean of `metric / max(metric)` across the filtered user set. It is recomputed client-side when the date range changes.
